Class 01 · Week 2
Designing the Knowledge Layer
Grounding an agent in enterprise data — accuracy, latency, trust, multi-source conflicts
Session score: 6.5 / 10
›
+ More classes coming

core tension
Zero tolerance for hallucination sounds right — but real-time, multi-source systems cannot guarantee perfect accuracy. The goal shifts from "eliminate all risk" to "manage risk transparently."
the framework — 5 steps
1
Name the impossibility first
Ask: can I guarantee zero error? Almost always no. Say so. Reframes from "eliminate risk" to "manage it transparently."
2
Scope the data sources
What data is needed? What's accessible? Freshness requirement per source? What happens when sources conflict?
3
Design three confidence modes
High: sources agree, fresh → answer with certainty. Medium: conflict or stale → answer + flag. Low: can't reconcile → say "I'm not sure."
4
Make uncertainty visible
Sources used, freshness, confidence level, what could be wrong. This isn't extra info — it's what makes trust possible.
5
Test against the user's real context
Where is the answer being used? In a live meeting? Async research? Acceptable for async is not acceptable mid-call.
5 questions to always ask
Scope
What subset of data can the user query? Own data only? Company-wide? Historical?
Freshness
How stale is too stale for this use case — minutes, hours, or days?
Conflict resolution
When sources disagree, what's the tiebreaker — timestamp, reliability, or user judgment?
Confidence expression
How do you communicate uncertainty without breaking the user's flow?
Fallback
What does the agent do when it genuinely cannot answer confidently?
how to read this
Realistic back-and-forth — candidate asks clarifying questions, states assumptions, builds incrementally. Annotations in italics explain why each move is made.
Round 1Opening question
Interviewer
You're building a Copilot feature for a sales rep querying their CRM in real time. The LLM sometimes hallucinates or gives outdated info. How do you think about the trade-off between accuracy, latency, and user trust?
Candidate
Before I get into trade-offs, can I clarify two things about the user and context?

First — is the rep doing async research before a call, or are they in an active meeting with a customer right now?

Second — scope of data: just their own deals, or company-wide including other reps and historical records?
💡Clarify user context before answering. The cost of being wrong — and therefore the accuracy bar — is completely different for async vs a live call.
Interviewer
They're on a live call. And they can only query their own deals.
Candidate
That changes everything. On a live call, a wrong data point in front of the customer is immediately costly — broken trust, bad negotiation. So accuracy wins over latency. A two-second wait for a right answer beats an instant wrong one.

On data: ten active deal records is small enough that a real-time REST API call at query time is cleaner and cheaper than a RAG pipeline. No stale index. API returns current state, LLM synthesises the answer.

One thing I'll name upfront: I'm not going to promise zero hallucination. That's not achievable in any live system. What I'll design for is transparent, managed uncertainty — so when something is wrong, the user can detect and correct it fast.
💡State the impossibility before the interviewer challenges it. Shows you've already thought past the naive answer.
Round 2Multi-source conflict
Interviewer
What if the query needs both CRM data and email history — like "is this deal actually on track?"
Candidate
One assumption before I answer: are emails linked to the deal record in the CRM, or is that a separate system call?

If linked — one API call gets both. If separate — I need to decide: parallel calls, or gate the email call behind CRM results? The answer changes the latency profile.
💡Always state the assumption before proceeding. Signals architecture thinking — you're thinking about what changes, not just the happy path.
Interviewer
Assume they're linked — one API call returns both.
Candidate
Good. One call, CRM and comms together, LLM synthesises across both.

Hard case: sources conflict. CRM says "closed won," latest email says negotiations still ongoing.

Timestamp is my first tiebreaker — most recent source wins. Email is more recent, so I treat the deal as open.

But I don't hide the conflict. The agent says: "Your CRM shows closed won, but your most recent email suggests negotiations are still open. I've used the email as the latest signal — please verify before acting on this."

One line. The rep reads it in a second. I don't ask them a question mid-call — I make the call and give them a clear path to override if I'm wrong.
💡Make the call, flag it in one line, let the user override. Don't ask a question mid-meeting — that breaks their flow.
Round 3Challenging "zero tolerance"
Interviewer
You said "zero tolerance for hallucination" but you just described picking a winner based on timestamp. Those aren't the same thing. How do you reconcile that?
Candidate
You're right — I should have drawn that distinction earlier.

"Zero tolerance for hallucination" means the agent never invents facts that don't exist in any source. It does not mean I can guarantee freshness or resolve every conflict perfectly — those are unavoidable in any live multi-source system.

The real design goal isn't zero error. It's: when an error occurs, the user can detect it, trace it, and correct it quickly. That means showing which sources I used and when they were last updated, surfacing a confidence level, and saying "I'm not sure" rather than guessing when confidence is genuinely low.

That's what trust means here — not that the AI is always right, but that the user always knows how much to rely on what they're getting.
💡When caught in a contradiction, own it and reframe with precision. Don't defend the earlier framing — upgrade it.
"Trust doesn't mean the AI is always right. It means the user always knows how much to rely on the answer — and can correct it fast when it's wrong."
Round 4Pushing on UX
Interviewer
You keep saying "surface the conflict to the user." The rep is in a meeting — they don't have time to read caveats. What does this actually look like?
Candidate
Fair — the UX can't be a wall of disclaimers.

Primary answer: confident and direct. "This deal is still in negotiation — last updated in your email two hours ago." That's what they say out loud.

The caveat is one inline line: "Note: CRM shows closed won. Email is more recent — I've used that." The rep reads it in a second, knows to confirm with their customer after the call.

The principle: answer is primary, caveat is a safety signal — not a disclaimer page. If we're showing more than one sentence of uncertainty framing, we've broken the UX.
💡When pushed on UX, give the actual output — what the user sees. Don't stay at the principle level. Show you've thought it to the interface.
mistake 1
Stating "zero tolerance for hallucination" without distinguishing it from stale data or source conflicts. Collapses under the first follow-up.
fix
Name the distinction early: "Zero tolerance means no invented facts — it doesn't mean I can guarantee freshness across live sources."
mistake 2
Jumping to architecture before establishing user context. The right architecture follows from who the user is and where they're using the answer.
fix
Always ask: who is the user, what are they doing, where, and what breaks if you're wrong. Architecture is a consequence of those answers.
mistake 3
Asking the user to verify after the call. They won't. And it doesn't solve the problem — they need the right answer now.
fix
Agent makes a call, states reasoning in one line, lets the user override inline. Don't defer to a future action that won't happen.
mistake 4
Saying "surface the conflict" without specifying what that looks like in the UI. Too abstract — doesn't show you've thought it to the interface level.
fix
Get concrete: primary answer is direct and confident, caveat is one inline sentence. Answer is primary. Caveat is a safety signal, not a disclaimer page.
